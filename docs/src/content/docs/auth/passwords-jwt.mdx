title: Passwords & JWT
description: Hash and verify passwords, sign and verify JWTs — with algorithm enforcement and constant-time compares.
---

The primitives are dependency-free and safe by construction.

## Passwords (scrypt)

```ts
import { hashPassword, verifyPassword } from "@mountsqli/auth";

const hash = await hashPassword("s3cret");
const ok = await verifyPassword(hash, "s3cret"); // true
```

- scrypt cost `N = 2^15` (above Node's `2^14` default).
- The cost factor is stored in the hash: `scrypt$<cost>$<salt>$<hash>`, so it
  can be raised later without breaking existing hashes.
- Comparison is **constant-time** (`timingSafeEqual`).

## JWT (HS256 + EdDSA)

```ts
import { signJwt, verifyJwt } from "@mountsqli/auth";

const token = await signJwt({ sub: "u1" }, secret, { alg: "HS256" });
const claims = await verifyJwt(token, secret);
```

EdDSA (Ed25519) is also supported via `generateEddsaKeys()`:

```ts
const { privateKey, publicKey } = generateEddsaKeys();
const t = await signJwt({ sub: "7" }, { key: privateKey, alg: "EdDSA" });
await verifyJwt(t, { key: publicKey, alg: "EdDSA" });
```

## Algorithm enforcement (closes confusion attacks)

`verifyJwt` **ignores the token's header `alg`** and enforces the configured one.
A token signed with a different algorithm is rejected with `reason: "bad-algorithm"`.

```ts
// Attacker forges an EdDSA token; server verifies HS256 → rejected.
const forged = signJwt({ sub: "attacker" }, { key: edKey, alg: "EdDSA" });
const v = await verifyJwt(forged, { key: "hS256-secret", alg: "HS256" });
// v.ok === false, v.reason === "bad-algorithm"
```

## API

| Function | Purpose |
| --- | --- |
| `hashPassword(pw)` | → scrypt hash string |
| `verifyPassword(hash, pw)` | → `boolean` (constant-time) |
| `signJwt(payload, key, opts)` | → token string |
| `verifyJwt(token, key, opts)` | → `{ ok, payload?, reason? }` |
| `generateEddsaKeys()` | → `{ privateKey, publicKey }` |

## Best practices

- Store the hash, never the password.
- Set `alg` explicitly on both sign and verify.
- Use a 32+ byte random secret for HS256.

## Common mistakes

- Letting the token's `alg` header decide verification (don't — MountSQLI blocks it).
- Reusing one secret across algorithms.
